# 数税云到聚兴 ERP 税务工作台：执行总控与验收手册

版本：2026-06-29

## 1. 总目标

把数税云中与聚兴 ERP「开票中心、发票反查业务、进项收票、留抵、金蝶核对、分析看板」相关的页面、接口、字段、动作和视觉结构完整迁移成 ERP 税务工作台。

最终闭环：

```text
数税云真实菜单 / 页面 / 接口
→ clone-website 视觉与结构提取
→ 静态 demo / 组件 spec
→ ERP 页面与业务映射
→ ERP 接口 adapter
→ 合同 / 出库 / 入库 / 金蝶 / 飞书审批 / 数税云融合
→ 灰度上线
→ 截图、接口、业务、税务风险验收
```

## 2. 最高口径

### 2.1 ERP 不是简单复制数税云

数税云负责：

- 真实税票数据源
- 税务平台页面和交互参考
- 开票、收票、查验、签收、勾选、认证、版式文件等税务能力

ERP 负责：

- 合同、出库、入库、客户、供应商、缺口台账
- 飞书审批
- 金蝶入账状态
- 税票和业务源单回挂
- 三方核对：ERP 业务事实、数税云税票、金蝶账务

### 2.2 开票主线必须固定

```text
待开票合同 gapToInvoice > 0
→ 我要开票
→ 创建 / 读取数税云真实开票草稿
→ 进入数税云真实开票页
→ 保存 / 预览
→ 提交飞书审批
→ 飞书审批通过
→ 系统调用数税云真实开票
→ 开票结果回填
→ 发票反查业务 / 金蝶入账核对
```

禁止把 ERP 旧 `invoice_application` 申请流程恢复为新主线。数税云「开票申请单」只做视觉和字段参考。

### 2.3 clone-website 的边界

clone-website 用于：

- 采集页面截图、DOM、CSS、资产
- 拆页面结构
- 生成组件 spec
- 生成静态 demo
- 做视觉和结构对照

clone-website 不用于：

- 直接接真实税务动作
- 直接写 ERP 主干业务逻辑
- 直接调用真实开票、红冲、勾选认证、签收等高风险接口
- 替代 ERP 数据模型设计

## 5. 风险分级

所有接口和按钮必须标风险等级。

| 等级 | 定义 | 示例 | 处理方式 |
|---|---|---|---|
| L0 | 纯查询 | 列表、详情、额度、字典 | 可优先接入 |
| L1 | ERP 本地状态 | 匹配、备注、本地标记 | 可接入，需审计 |
| L2 | 平台动作 | 同步、取票、下载、查验、任务重试 | 灰度接入 |
| L3 | 税务状态动作 | 签收、勾选、认证、入账提交、红字操作 | 必须单独审批和门禁 |
| L4 | 真实开票 / 红冲 | 真实开票、真实红冲 | 最高门禁，飞书审批后执行 |

L3 / L4 禁止在 demo 阶段直接触发真实动作。

## 7. 验收体系

### 7.0 机器验收命令

每轮推进后必须先跑机器验收，再写人工结论：

```bash
npm run tax-cloud:har:parse-all
npm run tax-cloud:audit
```

该命令会生成：

```text
docs/tax-cloud/TAX_CLOUD_VERIFICATION_REPORT.md
```

它会检查：

- 33 个目标页是否齐全。
- 每页 raw JSON、visible DOM、截图、design reference、page spec 是否齐全。
- 总控文档、缺口台账、动作矩阵、HAR 手册是否齐全。
- 非手工 P0 页面是否全部进入动作矩阵和 P0 demo。
- P1/P2/P3 页面是否进入动作矩阵。
- 每页 page spec 是否都有接口候选或明确接口策略。
- 每个非手工页面是否已经有 normalized HAR 真实接口证据。

严格完成验收必须跑：

```bash
npm run tax-cloud:audit:strict
```

`tax-cloud:audit:strict` 通过，才允许说“数税云采集、接口证据、demo 验收层已全部跑完”。如果 strict 失败，只能说当前处于部分完成状态，并以报告中的 FAIL 项作为下一步。

### 7.5 业务验收

必须跑通这些链路：

#### 链路 A：合同开票

```text
待开票合同
→ 我要开票
→ 草稿页预填
→ 保存 / 预览
→ 飞书审批
→ 审批通过
→ 真实开票
→ 发票回填
→ 金蝶入账核对
```

验收点：

- 合同金额、出库金额、票面金额一致。
- 客户资料正确预填。
- 商品、规格、单位、数量、单价、金额正确。
- 税编缺失时不能提交审批。
- 购方税号缺失时不能进入真实开票。

#### 链路 B：发票反查业务

```text
数税云发票池
→ 发票详情
→ 匹配合同 / 出库 / 入库
→ 判断金蝶是否入账
→ 异常进入队列
→ 人工处理 / 回挂 / 撤销
```

验收点：

- 发票能查到对应源单。
- 已入账 / 未入账清楚。
- 未匹配发票进入异常。
- 人工回挂可撤销。

#### 链路 C：进项收票

```text
进项发票池
→ 供应商 / 入库单匹配
→ 收票登记
→ 用途 / 勾选状态
→ 金蝶入账核对
```

验收点：

- 供应商识别正确。
- 入库单匹配正确。
- 勾选状态和认证结果可查。
- 金蝶入账差异可见。

#### 链路 D：分析看板

```text
发票池数据
→ 销项 / 进项 / 税率 / 票种 / 商品 / 地区 / 趋势
→ ERP 业务维度增强
```

验收点：

- 6 个分析看板独立存在。
- 图表和表格口径明确。
- 统计口径能追溯到发票明细。

## 12. 决策原则

1. 页面像不像，先看结构，再看 CSS。
2. 接口能不能接，先看风险，再看功能。
3. 数税云能做什么，不代表 ERP 要照搬什么。
4. ERP 业务事实优先于数税云页面流程。
5. 金蝶是账务状态中间层，数税云是税票源头，ERP 做三方关系判断。
6. 所有真实税务动作必须后置到审批和门禁之后。
